# Database Guide

This directory documents the application's data layer and provides a single,
assembled **Postgres schema contract** translated from the live **SQLite**
schema the app actually runs.

| Artifact | Purpose |
| --- | --- |
| [`schema.postgres.sql`](./schema.postgres.sql) | Full, ordered, runnable-looking Postgres DDL assembled from all modules. **Generated — do not hand-edit.** |
| [`erd.md`](./erd.md) | Mermaid entity-relationship diagrams grouped by domain. |
| [`modules/`](./modules/) | Per-module DDL (`*.sql`) + notes (`*.md`), one per backend module. |

**Source of truth:** the live SQLite schema created by each backend module's
`init(db)` function under `server/<module>/<module>.js`. The files here are a
faithful Postgres *translation* of that contract. When a module's `init(db)`
changes, regenerate the per-module DDL and re-assemble `schema.postgres.sql`.

**Current shape:** 79 tables, 136 foreign keys (+1 intentionally-disabled FK
preserved as a comment), 6 inferred UNIQUE constraints, 157 indexes.

---

## The no-migrations / `CREATE TABLE IF NOT EXISTS` pattern

The running app has **no migration framework**. Each module owns its tables and
creates them lazily at startup inside its `init(db)` function using
`CREATE TABLE IF NOT EXISTS ...` (and, in a few modules, ad-hoc `ALTER TABLE ...
ADD COLUMN` guarded by try/catch to add columns to already-existing tables).
Consequences that this contract preserves:

- **Idempotent DDL.** Every `CREATE TABLE`, `CREATE INDEX`, and (where natural)
  constraint is written so re-running the file is safe. The assembled Postgres
  file uses `CREATE TABLE IF NOT EXISTS` / `CREATE [UNIQUE] INDEX IF NOT EXISTS`
  for the same reason.
- **No ordered migration history.** There is no `schema_migrations` table and no
  up/down scripts. The schema *is* the union of the modules' `init(db)` output.
- **Order-independent foreign keys.** Because SQLite creates tables module-by-
  module in startup order, FKs cannot assume a target table already exists. In
  the Postgres contract we therefore emit **all** FKs as `ALTER TABLE ... ADD
  CONSTRAINT` in a dedicated section *after* every `CREATE TABLE`, so load order
  never breaks. (See section 2 of `schema.postgres.sql`.)
- **`master` and `report` own no tables.** They are read-only aggregators over
  other modules' tables — see [`modules/master.md`](./modules/master.md) and
  [`modules/report.md`](./modules/report.md).

> Migrating to Postgres for real would mean running section 1 (tables), then
> section 2 (FKs), then section 3 (indexes), once, in that order — and then
> introducing a proper migration tool going forward.

## SQLite → Postgres type mapping

SQLite is dynamically typed (type *affinity*), so columns store JS values
loosely: integers, ISO-8601 strings for dates/times, `0`/`1` for booleans, and
`REAL`/`NUMERIC` for numbers. The mapping below makes those intentions explicit
and **type-safe** in Postgres.

| SQLite (source) | Postgres (contract) | Notes |
| --- | --- | --- |
| `INTEGER PRIMARY KEY [AUTOINCREMENT]` | `BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY` | Surrogate keys. `BY DEFAULT` lets the app keep supplying explicit ids during data import. |
| `INTEGER PRIMARY KEY` on a 1:1 detail table (e.g. `company_gst_details.company_id`) | `BIGINT PRIMARY KEY` (no identity) + FK to parent | The PK **is** the FK; callers supply it. |
| `TEXT` | `TEXT` | No length caps in source → `TEXT`. Use `VARCHAR(n)` only where the source clearly caps length (none currently do). |
| `TEXT` holding ISO-8601 datetime (`created_at`, `updated_at`, `*_at`, `reconciled_at`) | `TIMESTAMPTZ` | App stores ISO strings via `datetime('now')`. Default → `DEFAULT now()`. |
| `TEXT` holding a date-only value (`voucher_date`, `report_date`, period start/end) | `DATE` | Date-only; no time component. |
| `TEXT` that is a free-form display string that *looks* like a date (e.g. `effective_date = '1-Apr-26'`) | `TEXT` | **Not** a real date — kept as `TEXT` (flagged in the module DDL). |
| `TEXT` holding serialized JSON | `JSONB` | e.g. `company_gst_details.state_wise_limits`. |
| `INTEGER` used as boolean (`0`/`1`, `is_*`, `has_*`, `show_*`, `enable_*`, `carry_*`, `*_enabled`) | `BOOLEAN` | **0 → `false`, 1 → `true`.** `DEFAULT 1/0` → `DEFAULT TRUE/FALSE`. |
| `INTEGER` that is a genuine count/ordinal (`display_order`, `minimum_hsn_length`) | `INTEGER` | Kept as integer. |
| `REAL`/`NUMERIC` holding a **currency amount** (`amount`, `debit`, `credit`, `value`, `balance`, `opening_balance`, `closing_balance`, threshold limits) | **`NUMERIC(18,2)`** | **Money is NEVER floating point.** See rationale below. |
| `REAL`/`NUMERIC` holding a **quantity or rate** (`qty`, `rate`, `gst_rate`, `max_days`) | **`NUMERIC(18,4)`** | Fixed-point, 4 dp for rates/quantities. |
| `REAL` that is a genuinely non-monetary free-form number (`company_feature_values.value_number`) | `DOUBLE PRECISION` | Only here is float acceptable. |

### Why money/quantity → `NUMERIC`, never floating point

This is an accounting application. `REAL`/`DOUBLE PRECISION` is IEEE-754 binary
floating point: `0.1 + 0.2 != 0.3`, and repeated rounding silently loses cents.
In a ledger that **breaks the fundamental Dr = Cr invariant** and makes balance
sheets, trial balances and GST totals fail to reconcile. Therefore:

- **Currency amounts → `NUMERIC(18,2)`** (exact, 2 decimal places).
- **Quantities & rates → `NUMERIC(18,4)`** (exact, 4 decimal places — rates and
  unit quantities need more precision than cents).
- `DOUBLE PRECISION` is used **only** for the rare non-monetary free-form number.

This contract flags **63 currency columns** as `NUMERIC(18,2)` and **86
quantity/rate columns** as `NUMERIC(18,4)`. Money columns flagged include, e.g.:
`ledgers.opening_balance` / `closing_balance` / `od_limit`,
`voucher_entries.amount`, voucher `cgst_amount` / `sgst_amount` / `igst_amount` /
`discount_amount` / `additional_amount` / `amount_forex`,
`stock_items.opening_rate` / `opening_value` / `rate` / `amount`,
the balance-sheet / P&L / trial-balance debit/credit/balance columns,
`company_gst_details.interstate_threshold_limit` / `intrastate_threshold_limit`,
`tcs_nature_of_goods.threshold_level`, and `tds_nature_of_payment.threshold_limit`.

### Why `0/1 INTEGER → BOOLEAN`

SQLite has no boolean type; the app stores `0`/`1`. Mapping these to a real
`BOOLEAN` (`0 → false`, `1 → true`) is self-documenting, prevents accidental
arithmetic on flags, and lets Postgres enforce the two-valued domain. `DEFAULT 1`
/ `DEFAULT 0` become `DEFAULT TRUE` / `DEFAULT FALSE`. When loading legacy SQLite
data, cast `0/1 → false/true`.

### Why ISO datetime `TEXT → TIMESTAMPTZ`

The app writes timestamps as ISO-8601 strings (via SQLite `datetime('now')`).
Storing them as `TIMESTAMPTZ` gives real temporal types — proper ordering, range
queries, and time-zone awareness — instead of lexical string comparison. The
SQLite default `datetime('now')` is translated to `DEFAULT now()`. Date-only
values (no time component, e.g. voucher dates) map to `DATE` instead.

## Foreign keys: explicit vs. inferred

SQLite frequently omits explicit `REFERENCES` clauses (and, by default, does not
even enforce the ones it has). Where the source **did** declare a FK, the
contract carries it through verbatim — including `ON DELETE CASCADE` /
`ON DELETE SET NULL` — and tags it `EXPLICIT in source`. Where the source only
had a `*_id` column (`company_id`, `fy_id`, `ledger_id`, `group_id`,
`parent_id`, `voucher_id`, `employee_id`, …), the relationship is **inferred**
and emitted as a proper Postgres FK, tagged accordingly. One FK
(`fk_voucher_type_configs_default_bank`) is left **disabled/commented** because
its target was unverified in the source; it is preserved for traceability.

All FKs live in **section 2** of `schema.postgres.sql` as `ALTER TABLE ... ADD
CONSTRAINT`, applied after every table exists, with a matching index per FK
column in **section 3**.

## Naming conventions

- **Tables:** `snake_case`, **plural** (`companies`, `ledgers`, `vouchers`).
  Child/line tables are prefixed by their parent (`voucher_entries`,
  `voucher_stock_entries`, `pay_head_slab_lines`, `balance_sheet_views`).
- **Primary keys:** singular `<entity>_id` (`ledger_id`, `voucher_id`), except a
  few line tables that use a generic `id` / `entry_id`.
- **Foreign keys (columns):** `<referenced_entity>_id` (`company_id`, `group_id`,
  `fy_id`). Self-references use `parent_*_id` (`parent_group_id`, `parent_id`).
- **Constraint names:** `fk_<child>_<purpose>`, `uq_<table>_<cols>`.
- **Index names:** `idx_<table>_<col>` (one per FK column), `uq_...` for unique.
- **Tenant column:** virtually every table carries `company_id`; data is
  multi-tenant and scoped per company, cascading on company delete.
- **Booleans:** `is_*`, `has_*`, `show_*`, `enable_*`, `*_enabled`.
- **Timestamps:** `created_at`, `updated_at`, and other `*_at` columns are
  `TIMESTAMPTZ`; pure dates use `*_date` / `date`.
