# Ledger Module — Database Schema

Source: `server/ledger/ledger.js` (SQLite). Postgres DDL: `docs/db/modules/ledger.sql`.

Three tables: `ledgers` (master), `ledger_bank_details` (1:1, optional),
`ledger_statutory_details` (1:1, optional).

General translation rules applied:
- `INTEGER PRIMARY KEY AUTOINCREMENT` -> `BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`.
- `*_balance`, `od_limit` -> **`NUMERIC(18,2)`** (money — never floating point).
- `*_rate`, `percentage_of_calculation`, `rounding_limit` -> **`NUMERIC(18,4)`** (rate/quantity precision).
- 0/1 flag columns -> **`BOOLEAN`** (`0`=false, `1`=true; defaults translated accordingly).
- `created_at` / `updated_at` (SQLite `datetime('now')` ISO strings) -> **`TIMESTAMPTZ DEFAULT now()`**.

## Notes

- **No UNIQUE constraints** exist in the SQLite source. Duplicate-active-name
  prevention is enforced in application code (`create`) via a case-insensitive
  query, not a DB constraint. A partial index `idx_ledgers_company_active_name`
  is provided to support that lookup; promote it to a partial UNIQUE index if you
  want DB-level enforcement.
- All FKs in this module are **explicit in the SQLite source** (`REFERENCES`
  clauses); none had to be inferred from `*_id` naming.
- The cross-module parent tables `companies` and `groups` are owned by other
  modules — their FK constraints are emitted here for completeness and are safe
  to run after all module tables exist.
