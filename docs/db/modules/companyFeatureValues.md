# companyFeatureValues — database schema

Source: `server/companyFeatureValues/companyFeatureValues.js`
Target: PostgreSQL (translated from SQLite)

This module defines a single table, `company_feature_values`, storing per-company values
for feature items defined in the `featureItem` module.

## Table: `company_feature_values`

| Column | Postgres type | Nullable | Default | Notes |
|---|---|---|---|---|
| `id` | `BIGINT GENERATED BY DEFAULT AS IDENTITY` | no | identity | PK. SQLite `INTEGER PRIMARY KEY AUTOINCREMENT`. |
| `company_id` | `BIGINT` | no | — | FK -> `companies.company_id` (explicit, ON DELETE CASCADE). |
| `feature_item_id` | `BIGINT` | no | — | FK -> `feature_items.feature_item_id` (explicit, ON DELETE CASCADE). |
| `value_boolean` | `BOOLEAN` | no | `false` | SQLite `INTEGER DEFAULT 0` used as 0/1 boolean. 0 => false, 1 => true. |
| `value_text` | `TEXT` | yes | — | Free-form text value for the feature. |
| `value_number` | `DOUBLE PRECISION` | yes | — | Generic numeric slot (SQLite `REAL`). No money/qty semantics in source; if a feature stores currency use `NUMERIC(18,2)`, if qty/rate use `NUMERIC(18,4)` — never floating point for money. |
| `value_date` | `DATE` | yes | — | SQLite `TEXT` holding a date-only string -> `DATE`. |
| `is_enabled` | `BOOLEAN` | no | `false` | SQLite `INTEGER DEFAULT 0` used as 0/1 boolean. 0 => false, 1 => true. |
| `created_at` | `TIMESTAMPTZ` | no | `now()` | SQLite `TEXT DEFAULT (datetime('now'))` ISO-8601 string -> `TIMESTAMPTZ`. |
| `updated_at` | `TIMESTAMPTZ` | no | `now()` | SQLite `TEXT DEFAULT (datetime('now'))`; service bumps it via `datetime('now')` on update. |

### Constraints & indexes

- `UNIQUE (company_id, feature_item_id)` — **inferred** (not in SQLite source). The
  service consistently selects/updates by this pair, so it is the logical key. Verify data
  before enforcing.
- Index on `company_id` (FK / primary lookup path — `get`, `update` queries).
- Index on `feature_item_id` (FK / join column).

## Relationships

- **`company_feature_values.company_id` -> `companies.company_id`** — explicit FK in source
  (`REFERENCES companies(company_id) ON DELETE CASCADE`). Many feature-value rows per company.
- **`company_feature_values.feature_item_id` -> `feature_items.feature_item_id`** — explicit
  FK in source (`REFERENCES feature_items(feature_item_id) ON DELETE CASCADE`). Each row
  corresponds to one feature item; joins expose `feature_key`, `feature_name`,
  `feature_group_id`.

## Money / numeric audit

- No currency columns in this table. `value_number` is a generic per-feature number mapped to
  `DOUBLE PRECISION`. Should any feature use this slot for monetary or quantity data, override
  the column (or per-feature handling) to `NUMERIC(18,2)` / `NUMERIC(18,4)` — floating-point
  types must never hold money.

## Seed data

On company creation the service seeds 19 rows (`feature_item_id` 1..19) with
`value_text`/`value_number`/`value_date` set NULL and `value_boolean`/`is_enabled` per the
`DEFAULT_VALUES` table in `companyFeatureValuesService.js` (items 1, 4, 5 enabled; the rest
disabled).
