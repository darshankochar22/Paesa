### Constraints / indexes
- PRIMARY KEY: `company_id`.
- No UNIQUE constraints in the source (note: `name` is **not** unique).
- No money/quantity columns in this table — no NUMERIC precision concerns here.

## Relationships

**Outgoing foreign keys:** none. `companies` is the top-level table.

**Incoming (inferred) foreign keys:** `companies.company_id` is referenced by
the `company_id` column of essentially every other module's tables — e.g.
`groups`, `ledgers`, `units`, `stock_groups`, `godowns`, `currencies`,
`voucher_types`, `gst_classifications`, `employee_categories`,
`employee_groups`, `payroll_units`, `financial_years`, `vouchers`, and more.
Those FK constraints are defined in the respective module SQL files and point
back to `companies (company_id)`. They are **inferred** from the `*_id` naming
convention (SQLite did not declare explicit FK constraints).

Recommended pattern (in each child module's SQL):

```sql
ALTER TABLE <child_table>
  ADD CONSTRAINT fk_<child>_company
  FOREIGN KEY (company_id) REFERENCES companies (company_id);  -- inferred
```
