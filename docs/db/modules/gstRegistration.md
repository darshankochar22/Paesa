Source: `server/gstRegistration/gstRegistration.js`

The SQLite schema is created via `CREATE TABLE IF NOT EXISTS`, plus a set of idempotent
`ALTER TABLE ... ADD COLUMN` migrations (for `address_type`, `goods_dispatched_from`,
`e_invoice_applicable_from`, `e_invoice_bill_from_place`, `composition_tax_rate`,
`composition_tax_calc_basis`). The Postgres translation below folds those migration columns into
the base table.

### Boolean conversion

`assessee_of_other_territory`, `e_invoice_application`, `e_way_bill_applicable`,
`applicable_for_intrastat`, and `is_active` are stored as `0`/`1` integers in SQLite and map to
Postgres `BOOLEAN` (`0 → false`, `1 → true`).

### Money / rate columns

`composition_tax_rate` is a precision-sensitive rate and is mapped to `NUMERIC(18,4)`, **not**
`REAL`/`DOUBLE PRECISION`. Never store money or tax rates in floating point.

## Relationships

- **`company_id` → `companies.company_id`** — *Explicit* foreign key in the source schema
  (`REFERENCES companies(company_id) ON DELETE CASCADE`). Indexed via
  `idx_gst_registrations_company_id`. Used by `getAll(company_id)`.

## Constraints & indexes

- Primary key: `gst_id`.
- Index: `idx_gst_registrations_company_id` on `company_id` (FK lookup support).
- Logical uniqueness of `(company_id, gstin)` among active rows is enforced **in application
  code only** (`create` rejects a duplicate active GSTIN). The SQLite schema has no UNIQUE
  constraint. An optional partial unique index is provided (commented out) in the `.sql` file.
