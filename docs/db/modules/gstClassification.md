# gstClassification module — database contract

Source: `server/gstClassification/gstClassification.js`
Postgres DDL: `docs/db/modules/gstClassification.sql`

This module owns one table: `gst_classifications`. The SQLite schema is created
once and then patched idempotently via `ALTER TABLE ... ADD COLUMN` for a set of
"required columns" (handling older databases). The Postgres translation below
reflects the full target schema.

## Table: `gst_classifications`

Per-company GST tax classification / rate profile.

### Indexes / constraints

- `idx_gst_classifications_company_id` — non-unique index on `company_id` (FK + the
  hot `WHERE company_id = ? AND is_active = 1` query path).
- `uq_gst_classifications_company_name_active` — **partial** unique index on
  `(company_id, lower(name)) WHERE is_active = true`. Mirrors the service-level
  case-insensitive duplicate check, which only considers active rows. A plain
  `UNIQUE` constraint cannot express the active-only predicate.

### Relationships

- `company_id` -> `companies.company_id` — **explicit FK** in the SQLite source
  (`REFERENCES companies(company_id) ON DELETE CASCADE`). Deleting a company
  cascades to its GST classifications.

No other `*_id` columns exist on this table, so no additional foreign keys are
inferred.

### Money / numeric handling

All rate columns (`igst_rate`, `cgst_rate`, `sgst_rate`, `cess_rate`, `gst_rate`)
are tax percentages and are mapped to `NUMERIC(18,4)`. They are intentionally
**not** floating-point: GST computations must be exact to avoid rounding drift in
invoices and returns.
