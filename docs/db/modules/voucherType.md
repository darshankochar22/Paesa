# voucherType module — database schema

Source: `server/voucherType/voucherType.js`. Two tables: `voucher_types` and `voucher_type_configs`. DDL (Postgres) in `voucherType.sql`.

Conventions used in the translation:
- SQLite `INTEGER PRIMARY KEY AUTOINCREMENT` -> `BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`.
- 0/1 flag columns -> `BOOLEAN` (0 -> `false`, 1 -> `true`); `DEFAULT 0/1` -> `DEFAULT false/true`.
- `created_at` / `updated_at` store ISO-8601 strings (`datetime('now')`) -> `TIMESTAMPTZ DEFAULT now()`.
- No money / quantity / rate columns in this module, so no `NUMERIC` mappings are needed.
- Columns added later via `ALTER TABLE ADD COLUMN` in the source carry the same defaults and are folded into the canonical `CREATE TABLE` here.

### Relationships — `voucher_types`
- `company_id` -> `companies.company_id` (EXPLICIT, ON DELETE CASCADE).
- `parent_vt_id` -> `voucher_types.vt_id` (EXPLICIT self-reference, ON DELETE SET NULL).
- Referenced by `voucher_type_configs.voucher_type_id` (one config row per voucher type).

### Relationships — `voucher_type_configs`
- `voucher_type_id` -> `voucher_types.vt_id` (EXPLICIT, ON DELETE CASCADE). Effectively a 1:1 extension of `voucher_types`.
- `default_bank_id` -> bank ledger account (INFERRED; no FK in source). Target table is owned by another module (ledgers/accounts) — FK left commented in the SQL pending confirmation of the referenced table/column.

## Notes
- The SQLite source has no `UNIQUE` constraints; the only uniqueness rule (active name per company, case-insensitive) is enforced in application code. A recommended partial unique index is included (commented) in `voucherType.sql`.
- All inferred-vs-explicit FK distinctions are annotated inline in `voucherType.sql`.
