# MVP — Developer & Agent Documentation

Onboarding index for the MVP codebase. This is the starting point for both future developers and AI agents working on the system.

## System overview

MVP is an Electron desktop application for accounting/inventory/payroll management (Tally-style). The **Electron main process** (`main.js`) owns the application lifecycle and hosts the backend; a **preload IPC bridge** (`preload.js`) exposes a typed `contextBridge` surface to the renderer; the **React/Vite renderer** (`client/`) is the UI; and persistence is **SQLite accessed through `@libsql/client`** in the backend (`server/`). Communication between renderer and backend is entirely over Electron IPC — roughly **252 IPC channels across 49 modules** — so the IPC channel set, not HTTP routes, is the real API contract of this app.

## The IPC contract

There is no HTTP layer — an IPC channel is the contract. For any single channel to work end to end, **three files must stay in sync**:

1. **`preload.js`** — declares the channel on the `contextBridge` and is what the renderer is actually allowed to call. If a channel isn't here, the renderer cannot reach it.
2. **`server/index.js`** — registers the `ipcMain` handler for the channel and implements (or dispatches) the backend logic.
3. **The frontend call site** (under `client/`) — invokes the channel and consumes its response shape.

When you add, rename, remove, or change the payload of a channel, update all three together — and update the matching docs (`docs/api/openapi.yaml`, the relevant `docs/api/modules/*`, and any affected DB docs). A channel that is present in one or two of these places but not all three is a latent bug.

## Status / scope

- These documents are **contracts and documentation derived from the current code** — a faithful snapshot of the IPC surface and SQLite schema as they exist now.
- **`docs/db/schema.postgres.sql` is a target contract, not an applied migration.** It is a Postgres translation of the current SQLite schema intended to describe the desired end state. No migration has been run, and the application still uses SQLite via `@libsql/client`.
- **No backend, renderer, or build code was changed** to produce this documentation. Nothing under `server/`, `client/`, `main.js`, `preload.js`, or any `package.json` was modified.

## How to regenerate

These docs are generated by reading the source of truth — `preload.js`, `server/index.js`, the backend modules, and the live SQLite schema — and re-deriving:

- the channel list and per-module groupings (the **Module index** above and `docs/api/modules/`),
- the OpenAPI spec (`docs/api/openapi.yaml`) from the request/response shapes of each channel,
- the Postgres target schema (`docs/db/schema.postgres.sql`), ERD (`docs/db/erd.md`), and DB guide from the current SQLite table definitions.

To refresh after code changes: re-extract the channels declared in `preload.js` and handled in `server/index.js`, re-derive each module's channel/table counts, regenerate the OpenAPI operations and DB DDL, and update this index. Keep the counts in the Module index consistent with the actual channels and tables in the code.
